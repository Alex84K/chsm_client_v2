# Инструкция по интеграции для фронтенд-разработчика: Панель администратора организации

В данном документе описано взаимодействие клиентского приложения (фронтенда) с API сервера для реализации следующих функций:
1. Авторизация администратора.
2. Получение списка доступных организаций для пользователя (выбор отделения/организации).
3. Получение списка участников организации.
4. Отправка приглашений новым пользователям.
5. Проверка и принятие приглашения новым пользователем.

---

## 1. Авторизация администратора (Login Flow)

Администратор организации авторизуется с помощью глобальной формы входа.

### API-запрос
* **Метод**: `POST`
* **Путь**: `/auth/login`
* **Заголовки**: 
  - `Content-Type: application/json`

#### Тело запроса (Request Body):
```typescript
interface LoginRequest {
  email: string;      // Email администратора
  password: string;   // Пароль администратора
}
```

### API-ответ (Response)
* **Статус**: `200 OK` (при успехе) или `401 Unauthorized` (при неверных данных)

#### Тело ответа (Response Body):
```typescript
interface LoginResponse {
  accessToken: string; // JWT токен сессии (срок действия 24 часа)
  user: {
    id: string;        // ID пользователя
    email: string;     // Email
    name: string | null;
    globalRole: 'USER' | 'ADMIN' | 'SUPERADMIN' | 'TEACHER' | 'STUDENT' | 'MANAGER';
  }
}
```

### UX-логика:
1. После успешного запроса сохраните `accessToken` в `localStorage` (или в безопасных http-only cookies, если применимо).
2. Для всех последующих запросов, требующих авторизации, передавайте токен в заголовке `Authorization: Bearer <accessToken>`.
3. Сразу после входа необходимо получить список доступных пользователю организаций (см. раздел 1.5).

---

## 1.5. Получение доступных организаций для пользователя (Choose Organization)

После авторизации пользователь (администратор, преподаватель или студент) может состоять в одной или нескольких организациях. Фронтенд должен запросить список доступных для него организаций, чтобы пользователь мог выбрать нужную организацию (отделение).

### API-запрос (Рекомендуемый способ через токен)
* **Метод**: `GET`
* **Путь**: `/auth/user/organizations`
* **Заголовки**:
  - `Authorization: Bearer <accessToken>`
  - `Content-Type: application/json`

### Альтернативный API-запрос (по ID пользователя)
* **Метод**: `GET`
* **Путь**: `/auth/user/:userId/organizations`
  * Где `:userId` — это UUID пользователя (из ответа login/activate).
  * Доступно только самому пользователю (сверяется по токену) либо глобальному `SUPERADMIN`.
* **Заголовки**:
  - `Authorization: Bearer <accessToken>`
  - `Content-Type: application/json`

### API-ответ (Response)
* **Статус**: `200 OK` (успешно)

#### Тело ответа (Response Body):
```typescript
interface UserOrganizationItem {
  id: string;       // ID организации (organizationId)
  name: string;     // Название организации
  slug: string;     // URL-slug организации
  role: 'USER' | 'ADMIN' | 'TEACHER' | 'STUDENT' | 'MANAGER'; // Роль пользователя в данной организации
  joinedAt: string; // Дата вступления
}

type UserOrganizationsResponse = UserOrganizationItem[];
```

### UX-логика:
1. Сделайте запрос `GET /auth/user/organizations` сразу после входа пользователя в приложение.
2. **Если вернулась ровно 1 организация**:
   - Автоматически сохраните её `id` как активную организацию в стейте (или `sessionStorage`).
   - Перенаправьте пользователя на соответствующий дашборд.
3. **Если вернулось более 1 организации**:
   - Покажите пользователю промежуточный экран выбора («Выберите отделение / организацию для входа»).
   - Выведите карточки или список с названиями организаций и ролями пользователя в них.
   - При выборе сохраните `id` организации и перенаправьте пользователя.
4. **Если организаций нет**:
   - Покажите сообщение: *"Вы не состоите ни в одной организации. Пожалуйста, обратитесь к администратору за приглашением или активируйте новую организацию"*.

## 2. Получение списка участников организации (Members List)

Позволяет загрузить и отобразить таблицу всех текущих участников конкретной организации.

### API-запрос
* **Метод**: `GET`
* **Путь**: `/organizations/:id/members`
  * Где `:id` — это UUID организации (например, `4bb22550-a17f-495d-beec-5b789eb22f3e`).
* **Заголовки**:
  - `Authorization: Bearer <accessToken>`
  - `Content-Type: application/json`

### API-ответ (Response)
* **Статус**: `200 OK` (успех), `401 Unauthorized` (токен истек/отсутствует), `403 Forbidden` (если текущий пользователь не является `ADMIN` или `TEACHER` в данной организации).

#### Тело ответа (Response Body):
```typescript
interface OrgMemberListItem {
  id: string;            // ID записи о членстве
  userId: string;        // ID пользователя
  role: 'USER' | 'ADMIN' | 'TEACHER' | 'STUDENT' | 'MANAGER'; // Роль внутри организации
  createdAt: string;     // Дата добавления в организацию
  user: {
    id: string;
    email: string;
    name: string | null;
    globalRole: string;
  }
}

type GetMembersResponse = OrgMemberListItem[];
```

### UX-логика:
1. Выполняйте запрос при входе администратора в раздел "Участники".
2. Отображайте список в виде таблицы: Имя, Email, Локальная роль, Дата добавления.
3. Если сервер возвращает `403 Forbidden`, заблокируйте интерфейс управления пользователями и покажите сообщение: *"У вас недостаточно прав для просмотра списка участников данной организации"*.

## 3. Рассылка приглашений новым пользователям (Send Invitation)

Позволяет администратору организации отправить приглашение на email нового пользователя. На указанный адрес будет отправлено письмо со ссылкой-приглашением.

### API-запрос
* **Метод**: `POST`
* **Путь**: `/organizations/:id/invitations`
  * Где `:id` — это UUID организации.
* **Заголовки**:
  - `Authorization: Bearer <accessToken>`
  - `Content-Type: application/json`

#### Тело запроса (Request Body):
```typescript
interface SendInvitationRequest {
  email: string; // Email приглашаемого пользователя
  role: 'STUDENT' | 'USER'; // Локальная роль, которую получит пользователь (разрешены только STUDENT или USER)
}
```

### API-ответ (Response)
* **Статус**: `201 Created` (успешно отправлено), `400 Bad Request` (невалидный email или некорректная роль), `403 Forbidden` (приглашать могут только `ADMIN` или `TEACHER` организации), `409 Conflict` (если для этого email в эту организацию уже есть активное неотклоненное приглашение).

#### Тело ответа (Response Body):
```typescript
interface InvitationResponse {
  id: string;             // ID приглашения
  email: string;          // Email получателя
  role: 'STUDENT' | 'USER';
  organizationId: string;
  status: 'PENDING' | 'ACCEPTED';
  expiresAt: string;      // Дата/время истечения ссылки (48 часов с момента создания)
}
```

### UX-логика:
1. Добавьте модальное окно "Пригласить участника" с полями:
   - Email (валидация на клиенте).
   - Выпадающий список выбора роли (`STUDENT` или `USER`). По умолчанию выставляйте `STUDENT`.
2. Кнопка отправки должна переходить в состояние `loading` во время выполнения запроса.
3. При успешном получении ответа `201 Created` покажите Toast/Alert: *"Приглашение успешно отправлено на <email>. Ссылка действительна 48 часов."* и обновите список участников (или список отправленных приглашений, если он выведен отдельно).
4. Обрабатывайте ошибку `409 Conflict`: выведите сообщение *"На этот email уже отправлено активное приглашение в эту организацию"*.

## 4. Проверка и принятие приглашения (Verification & Acceptance Flow)

Этот сценарий выполняется приглашенным пользователем, который переходит по ссылке из email. Ссылка на фронтенд должна иметь формат вроде `https://front.domain.com/invitation?token=<token>`.

### Шаг A: Проверка токена (Verify Token)
Вызывается автоматически при загрузке страницы принятия приглашения для проверки валидности токена и получения контекста.

* **Метод**: `GET`
* **Путь**: `/auth/invitations/verify?token=<token_из_url>`
* **Авторизация**: Не требуется (публичный эндпоинт).

#### Ответ при успехе (200 OK):
```typescript
interface VerifyInvitationResponse {
  email: string;            // Email, на который было отправлено приглашение (для предзаполнения формы)
  organizationName: string; // Имя организации, в которую приглашают (для отображения на приветственном экране)
  role: 'STUDENT' | 'USER'; // Будущая роль пользователя
}
```
* **Если токен не найден**: Вернется `404 Not Found`.
* **Если токен истек (>48 часов) или уже принят**: Вернется `400 Bad Request` с соответствующим сообщением.

#### UX-логика проверки:
1. При открытии страницы покажите Loader.
2. Отправьте запрос на `/auth/invitations/verify`.
3. При успешном ответе:
   - Скройте Loader.
   - Отобразите форму завершения регистрации с текстом: *"Вы приглашены в организацию <organizationName> в качестве <role>"*.
   - Заблокируйте поле Email и автоматически вставьте туда `email` из ответа.
   - Покажите обязательное поле ввода пароля (минимум 8 символов) и опционально — поле ввода Имени.
4. При ошибке (400/404):
   - Заблокируйте форму и покажите крупный Alert: *"Ссылка недействительна или срок её действия истек. Запросите новое приглашение у администратора организации."*

---

### Шаг B: Завершение регистрации и вход (Accept Invitation)
Отправляется пользователем после заполнения пароля.

* **Метод**: `POST`
* **Путь**: `/auth/invitations/accept`
* **Авторизация**: Не требуется (публичный эндпоинт).

#### Тело запроса (Request Body):
```typescript
interface AcceptInvitationRequest {
  token: string;          // Токен приглашения из URL
  password: string;       // Пароль пользователя (минимум 8 символов)
  name?: string;          // Опциональное имя пользователя
}
```

#### Ответ при успехе (201 Created):
Сервер автоматически создает/обновляет пользователя, привязывает его к организации, помечает токен как принятый и сразу возвращает JWT-сессию для входа.
```typescript
interface AcceptInvitationResponse {
  accessToken: string; // Сразу авторизуем пользователя
  user: {
    id: string;
    email: string;
    name: string | null;
    globalRole: string;
  }
}
```

#### UX-логика принятия:
1. После успешного запроса сохраните полученный `accessToken` в `localStorage` (выполните стандартный флоу логина).
2. Перенаправьте пользователя на дашборд его новой организации. Поздравьте с успешной регистрацией.
