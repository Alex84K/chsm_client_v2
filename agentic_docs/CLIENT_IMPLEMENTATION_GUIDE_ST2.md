# Руководство по реализации клиентского приложения (Client Integration Guide - Stage 4)
## Модуль: Успеваемость (Enrollments, Grades, Gradebook)

Данный документ представляет собой подробную спецификацию и практическое руководство для Frontend-агента (разработчика), который будет реализовывать интерфейс зачисления студентов, выставления оценок и утверждения ведомостей в **Student Portal SPA** и **Admin Panel SPA**.

---

## 1. Доменные типы TypeScript (Types)

Frontend-приложение должно использовать следующие типы и перечисления для синхронизации с бэкендом:

### 1.1 Перечисления (Enums)
- **EnrollmentStatus** (Статус зачисления студента на поток)
  - `ENROLLED` — студент зачислен и активно обучается на потоке.
  - `COMPLETED` — студент успешно завершил обучение на данном потоке.
  - `DROPPED` — студент отчислен или покинул поток до его завершения.
- **GradebookStatus** (Статус зачетной ведомости зачисления)
  - `DRAFT` — черновик, оценки активно выставляются и редактируются преподавателем.
  - `SUBMITTED` — ведомость отправлена куратором на проверку администрации (оценки заморожены для куратора).
  - `APPROVED` — ведомость утверждена администрацией. Оценки полностью заблокированы для любых изменений на бэкенде.

### 1.2 Интерфейсы сущностей (Interfaces)
- **GradeEntry** (Оценка по конкретному предмету)
  - `id`: string (UUID) — уникальный ID оценки.
  - `subjectId`: string (UUID) — ID предмета из `Subject`.
  - `value`: number — полученный балл (целое число >= 0).
  - `source`: string | null — источник оценки (например, "Classroom", "Exam").
  - `recordedAt`: string (ISO Date) — дата выставления/изменения оценки.
- **Gradebook** (Состояние ведомости зачисления)
  - `id`: string (UUID).
  - `status`: GradebookStatus — статус ведомости.
  - `approvedAt`: string | null (ISO Date) — дата утверждения ведомости администрацией.
  - `approvedBy`: string | null — имя ответственного администратора, подписавшего ведомость.
- **Enrollment** (Зачисление студента на поток — Корень Агрегата)
  - `id`: string (UUID) — ID зачисления.
  - `studentId`: string (UUID) — ID карточки студента (`Student`).
  - `sessionRunId`: string (UUID) — ID потока (`SessionRun`), на который зачислили.
  - `status`: EnrollmentStatus — статус зачисления.
  - `enrolledAt`: string (ISO Date) — дата зачисления на поток.
  - `approvedAt`: string | null (ISO Date) — дата перевода в статус COMPLETED.
  - `approvedBy`: string | null — имя администратора, утвердившего завершение обучения.
  - `teacherName`: string | null — имя закрепленного куратора/преподавателя.
  - `gradebook`: Gradebook — вложенный объект ведомости.
  - `grades`: GradeEntry[] — массив выставленных оценок по предметам.

## 2. Спецификация API (Endpoints)

Все запросы требуют заголовок `Authorization: Bearer <JWT_TOKEN>`. URL содержат префикс тенанта `:orgId` для изоляции данных.

### 2.1 Управление зачислениями (Enrollments)
- **POST** `/organizations/:orgId/enrollments` — Зачислить студента на поток.
  - *Payload:* `{ "studentId": "uuid", "sessionRunId": "uuid", "teacherName": "Имя преподавателя" (опц) }`
  - *Response (201 Created):* Объект `Enrollment`
- **GET** `/organizations/:orgId/enrollments` — Получить список всех зачислений организации.
  - *Response (200 OK):* Массив `Enrollment[]`
- **GET** `/organizations/:orgId/enrollments/:id` — Получить детальную информацию о зачислении по ID (включая ведомость и оценки).
  - *Response (200 OK):* Объект `Enrollment`
- **DELETE** `/organizations/:orgId/enrollments/:id` — Удалить (аннулировать) зачисление.
  - *Response (204 No Content)*
- **PATCH** `/organizations/:orgId/enrollments/:id/status` — Изменить статус зачисления (например, завершить обучение).
  - *Payload:* `{ "status": "COMPLETED" | "DROPPED", "approvedBy": "Имя админа" (опц) }`
  - *Response (200 OK):* Обновленный объект `Enrollment`

### 2.2 Выставление и редактирование оценок (Grades)
- **POST** `/organizations/:orgId/enrollments/:id/grades` — Выставить или обновить оценку за предмет.
  - *Payload:* `{ "subjectId": "uuid", "value": 5 }`
  - *Response (201 Created):* Обновленный объект `Enrollment` (с добавленной/измененной оценкой в массиве `grades`).
- **PATCH** `/organizations/:orgId/enrollments/:id/grades/:gradeId` — Изменить существующую оценку.
  - *Payload:* `{ "newValue": 4 }`
  - *Response (200 OK):* Обновленный объект `Enrollment`
- **DELETE** `/organizations/:orgId/enrollments/:id/grades/:gradeId` — Удалить оценку за предмет.
  - *Response (204 No Content)*

### 2.3 Жизненный цикл ведомости (Gradebook Workflow)
- **POST** `/organizations/:orgId/enrollments/:id/gradebook/submit` — Отправить ведомость куратором на проверку администрации.
  - *Response (200 OK):* Обновленный объект `Enrollment` (статус ведомости изменится на `SUBMITTED`).
- **POST** `/organizations/:orgId/enrollments/:id/gradebook/approve` — Утвердить ведомость (подпись администратора, блокирующая редактирование).
  - *Payload:* `{ "approvedBy": "Директор / Завуч" }`
  - *Response (200 OK):* Обновленный объект `Enrollment` (статус ведомости изменится на `APPROVED`).

## 3. Проектирование интерфейса и UX-сценарии (UX Guidelines)

Реализация этих сценариев на клиенте требует тщательной проработки UX для минимизации ошибок пользователей и наглядного отображения бизнес-логики.

### 3.1 Сценарий 1: Интерактивная «Зачетная ведомость» (Grading Matrix)
Вместо разрозненных форм рекомендуется сделать один главный экран — **Электронный журнал потока**.
- **Визуализация:** Таблица, где строки — это студенты (зачисления `Enrollment`), а колонки — предметы (`Subject`), привязанные к этому потоку.
- **Ячейка таблицы:** Отображает текущую оценку студента по предмету. 
  - При клике на ячейку она превращается в поле ввода (Input) или выпадающий список (Dropdown) со шкалой оценивания.
  - Рядом с оценкой в подсказке (Tooltip) выводить шкалу предмета: например, `4 / 5` (где 5 — `scale` предмета).
- **Валидация на лету:** Поле ввода должно иметь ограничение `max` равное значению `scale` этого предмета. Если преподаватель вводит `6` при шкале `5`, кнопка сохранения блокируется, а поле подсвечивается красной ошибкой: *«Максимальная оценка по этому предмету: 5»*.

### 3.2 Сценарий 2: Механизм блокировки оценок (Gradebook Lock UX)
Оценки в ячейках должны динамически реагировать на статус зачетки `Enrollment.gradebook.status`.
- **Если статус ведомости `APPROVED` или `SUBMITTED`:**
  - Все ячейки оценок этого студента переводятся в режим **Read-Only** (некликабельные текстовые поля).
  - В углу карточки студента или строки таблицы отображается иконка «Замка» 🔒.
  - При наведении на замочек выводится подсказка: *«Оценки заблокированы, так как ведомость утверждена / отправлена на проверку»*.
- **Если статус ведомости `DRAFT`:**
  - Отображается иконка «Карандаша» ✏️, сигнализирующая о возможности редактирования.

### 3.3 Сценарий 3: Управление ведомостью (Gradebook Workflow Controls)
В панели управления успеваемостью студента (или куратора) должен быть блок действий над ведомостью:
- **Роль куратора (статус `DRAFT`):** Отображается кнопка **«Отправить ведомость на подпись»** (вызывает `/gradebook/submit`). После нажатия интерфейс блокируется для редактирования куратором.
- **Роль куратора (статус `SUBMITTED`):** Отображается плашка: *«Ведомость отправлена на проверку администрации. Ожидайте утверждения»*.
- **Роль администратора (статус `SUBMITTED`):** Отображаются две кнопки:
  - **«Утвердить ведомость»** (вызывает `/gradebook/approve` с вводом имени админа). Ведомость переходит в статус `APPROVED`.
  - **«Вернуть на доработку»** (кнопка, которая переводит ведомость обратно в `DRAFT` для разблокировки оценок куратору. *Реализуется через бизнес-команду бэкенда, куратор снова сможет вносить правки*).

### 3.4 Сценарий 4: Завершение обучения на потоке (Graduation UX)
Когда поток подходит к концу, администратор может перевести зачисление студента в статус `COMPLETED` (Обучение завершено).
- **UX-Инвариант:** Кнопка «Завершить обучение студента» должна быть **неактивна (Disabled)**, если статус ведомости студента (`Enrollment.gradebook.status`) равен `DRAFT` или `SUBMITTED`.
- При наведении на неактивную кнопку выводить подсказку: *«Для завершения обучения ведомость успеваемости должна быть утверждена администрацией (статус APPROVED)»*.
- После успешного утверждения ведомости кнопка становится активной. Нажатие на нее вызывает `PATCH /enrollments/:id/status` с телом `{ "status": "COMPLETED", "approvedBy": "Имя директора" }`.

## 4. Обработка ошибок бэкенда на клиенте

Бэкенд возвращает стандартные HTTP-ошибки с понятными сообщениями. Фронтенд должен маппить их в человекочитаемые алерты (Alerts):

- **Ошибка 409 (Conflict) при зачислении:**
  - *Причина:* Попытка зачислить студента на поток, где он уже учится.
  - *UI Alert:* *«Данный студент уже зачислен на этот учебный поток. Повторное зачисление невозможно»*.
- **Ошибка 409 (Conflict) при выставлении оценки:**
  - *Причина:* Предмет принадлежит другому потоку.
  - *UI Alert:* *«Ошибка консистентности: этот предмет не преподается в рамках текущего потока»*.
- **Ошибка 400 (Bad Request) со статус-кодом заблокированной ведомости:**
  - *Причина:* Попытка изменить оценку при статусе ведомости `APPROVED`.
  - *UI Alert:* *«Действие заблокировано. Оценки в утвержденной ведомости не могут быть изменены»*.
- **Ошибка 404 (Not Found) при Tenant Isolation:**
  - *Причина:* Студент или Поток принадлежат другой школе (попытка подмены ID).
  - *UI Alert:* *«Запрашиваемый ресурс не найден в вашей организации»*.
