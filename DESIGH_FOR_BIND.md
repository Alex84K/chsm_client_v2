# Проектирование (Design): Вложенный интерфейс "Запуски сессий" (Session Runs Binding)

Данный документ описывает архитектурное решение для связывания учебных потоков (Session Runs) со Студентами и Предметами на уровне клиентского приложения. Дизайн базируется на паттерне "Client-side JOIN" и использовании MUI Collapsible Rows, опираясь на логику из старой системы.

---

## 1. Слой данных и API (Data Layer)

Для обеспечения данных во вложенных вкладках требуется загрузить весь массив зачислений организации и сопоставить его с сущностями на клиенте.

### 1.1 Новые эндпоинты и хуки
- **Эндпоинт:** Добавить в `src/api/enrollments.api.ts`:
  ```typescript
  export const getAllOrganizationEnrollments = async (orgId: string): Promise<Enrollment[]> => { ... }
  ```
- **Хук:** Добавить в `src/hooks/useEnrollments.ts`:
  ```typescript
  export const useAllOrganizationEnrollments = (orgId: string) => useQuery(...)
  ```

### 1.2 Агрегация данных (Client-side JOIN)
Для каждой строки потока (`SessionRun`) данные вычисляются "на лету" с помощью `useMemo` или простых фильтров в компоненте строки:
- **Предметы:** `allSubjects.filter(sub => sub.sessionRunId === run.id)`
- **Зачисления (Enrollments):** `allEnrollments.filter(e => e.sessionRunId === run.id && e.status !== 'DROPPED')`
- **Студенты:** Агрегируются из `allStudents` по совпадению `studentId` из отфильтрованных `runEnrollments`.

## 2. Компонентная структура (UI Architecture)

Текущий компонент `SessionsList.tsx` слишком велик. Необходимо вынести логику рендеринга одной строки потока в отдельный компонент для инкапсуляции состояния (открыто/закрыто, активная вкладка).

### 2.1 Иерархия компонентов
1. `SessionsList` (Контейнер, загружающий все хуки: `useSessionRuns`, `useSubjects`, `useStudents`, `useAllOrganizationEnrollments`).
2. `SessionRunRow` (Новый компонент для одной строки потока).
   - *State:* `open` (boolean) — для управления `<Collapse>`.
   - *State:* `tabIndex` (number: 0 | 1) — управление вкладками (0 = Студенты, 1 = Предметы).
3. `<Collapse>`
   - `<Tabs>` (СТУДЕНТЫ | ПРЕДМЕТЫ)
   - **TabPanel 0 (Студенты):**
     - Таблица/Список с ФИО, специальностью, Telegram.
     - Кнопка "+ ДОБАВИТЬ СТУДЕНТА".
   - **TabPanel 1 (Предметы):**
     - Таблица/Список предметов (Название, часы, интеграция Classroom).
     - Кнопка "+ ДОБАВИТЬ ПРЕДМЕТ".

## 3. Модальные окна и Формы (Modals)

Так как контекст добавления находится внутри конкретного потока (`sessionRunId` жестко задан), мы создаем **новые** специфичные модальные окна, не модифицируя базовые модалки управления ресурсами.

### 3.1 Модалка добавления студента на поток (`EnrollStudentToRunModal.tsx`)
Полностью реализует логику зачисления в контексте сессии, опираясь на старую систему:
- **Пропсы:** `open`, `onClose`, `defaultSessionRunId` (идентификатор текущего потока).
- **Поля:**
  - `studentId` — выпадающий список всех студентов.
  - `enrolledAt` — дата зачисления.
  - `status` — выбор статуса (`ENROLLED`, `DROPPED`, `COMPLETED`).
- **Валидация бизнес-правил:**
  - Проверка: студент **не может** быть зачислен на другую сессию в рамках **того же** учебного года (`academicYearId`). Если такое зачисление найдено, форма выдает ошибку.
- Вызывает `createEnrollmentMutation`.

### 3.2 Модалка добавления предмета в поток (`AddSubjectInSessionModal.tsx`)
Специальная модалка для добавления предметов, где `sessionRunId` скрыт и подставляется автоматически:
- **Пропсы:** `open`, `onClose`, `defaultSessionRunId`.
- **Поля:**
  - `title` — **Autocomplete** по уникальным названиям уже существующих предметов (`Array.from(new Set(allSubjects.map(s => s.title)))`). Если выбрать существующее название, форма автоматически предзаполняет часы (`hours`) и признак интеграции (`hasClassroom`).
  - `teacherName` — текстовое поле.
  - `hours` — количество часов.
  - `scale` — вес предмета (1-3).
  - `isCore` — блокирующий предмет (Switch).
  - `hasClassroom` и `classroomCourseworkId` — интеграция с Google Classroom.
- Вызывает `createSubjectMutation`. Базовый `SubjectModal.tsx` остается нетронутым.

## 4. Сценарии действий (User Flows)

1. **Добавление студента:** Пользователь жмет "+ ДОБАВИТЬ СТУДЕНТА" во вкладке открытого потока. В новой модалке выбирает студента (с проверкой на уникальность в учебном году) -> Создается `Enrollment` -> Таблица студентов потока обновляется.
2. **Удаление студента с потока:** Нажатие на иконку 🗑️ в строке студента.
   - Вызывает `updateEnrollmentStatus` со статусом `DROPPED` или `deleteEnrollment`.
3. **Добавление предмета:** Пользователь жмет "+ ДОБАВИТЬ ПРЕДМЕТ". Открывается `AddSubjectInSessionModal`, пользователь вводит название (или выбирает из `Autocomplete`) -> Создается предмет, жестко привязанный к открытому потоку.
4. **Кнопка "ОЦЕНКИ":** На строке предмета расположена кнопка "ОЦЕНКИ" — она перенаправляет администратора либо в профиль предмета, либо в матрицу оценок ведомости.
